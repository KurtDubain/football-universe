# 上帝视角观察玩法与参赛体验清单

创建日期：2026-07-22

这份清单负责《足球联赛宇宙》下一阶段的玩法、故事、操作反馈和参赛体验打磨。它不替代已有的 `data-chain-optimization-checklist.md` 与 `ui-polish-checklist.md`：

- 数据真实性、统计一致性、长期存档和性能预算继续由数据链路清单约束。
- 通用视觉规范、移动端基础体验和资产预算继续由 UI 清单约束。
- 本清单只追踪“观察者为什么在意、能做什么、世界如何回应、历史如何沉淀”。

**勾选规则：实现不等于完成。只有功能、数据语义、自动化测试、桌面/移动端浏览器体验和性能验证全部通过后，才能将条目标记为完成。**

## 0. 已冻结的产品方向

- [x] 玩家身份是足球宇宙的上帝视角观察者和历史见证者，不是经理或教练。
- [x] 核心乐趣是观察球队、球员、教练、赛事和王朝随时间自然演化。
- [x] 玩家可以选择关注对象、做出判断、控制时间和回看历史，但不负责阵容、战术、训练和日常经营。
- [x] 所有关注、预测、故事和展示层功能不得暗中改变比赛结果。
- [x] 比赛结果继续由可复现的种子模拟产生；解释层必须引用真实模型因素或真实事件，不得编造因果。
- [x] 默认球队和教练名称当前保持不变，继续服务个人游玩体验。
- [x] 自定义球队入口继续保留，不因参赛体验改造而隐藏或降级。
- [x] 投稿所需原创名称与资产未来通过独立的“参赛宇宙预设”处理，不复制模拟引擎，也不覆盖个人经典宇宙。
- [x] 不引入后端、强制联网、付费系统或 AI 运行时依赖。

## 1. 目标体验与验收基线

### 1.1 核心循环

- [ ] 将每个关键轮次稳定收束为：`关注故事 -> 赛前判断 -> 观看/揭晓 -> 理解结果 -> 故事与历史推进`。
- [ ] 不观看直播的玩家也能在一次普通推进后获得完整的结果、解释和故事反馈。
- [ ] 观看直播的玩家获得更强氛围和事件过程，但不会获得影响赛果的额外权限。
- [ ] 每个首屏最多保留一个主要操作，其他功能保持次级，不与推进争夺注意力。
- [ ] 普通轮次可以快速完成，关键比赛才使用完整动画与更强叙事节奏。

### 1.2 前五分钟目标

- [x] 新玩家在 30 秒内理解“我不操控球队，我观察一个自动演化的足球世界”。
- [x] 新玩家在 60 秒内进入宇宙，并完成关注球队或“纯观察”选择。
- [x] 新玩家在 2 分钟内看到一场明确的焦点比赛、赛前概率和真实影响因素。
- [x] 新玩家在 3 分钟内完成一次观察判断并看到比赛结果。
- [x] 新玩家在 4 分钟内理解预测为何命中或落空，以及比赛是否属于爆冷。
- [ ] 新玩家在 5 分钟内看到至少一条故事或观察主题发生变化。
- [ ] 首次体验不依赖玩家主动寻找隐藏页面，也不要求阅读长篇教程。

### 1.3 十五分钟容量

- [ ] 设计一条可重复验证的 15 分钟评审路线，覆盖至少两个普通轮次、一个关键比赛和一次阶段性故事更新。
- [ ] 15 分钟内让玩家看到球队、球员或教练至少一个可感知的状态变化。
- [ ] 15 分钟结束时提供可继续游玩的悬念，而不是一次性演示结束页。
- [ ] 推荐体验和自由长期游戏共用同一套真实模拟规则，不为演示伪造固定结果。

## 2. P0 现有观察链路正确性

### 2.1 观察判断结算统一

- [x] 抽取单一的观察判断结算函数，禁止 `advanceWindow`、`batchAdvance` 和 `advanceUntil` 各自实现不同语义。
- [x] 普通推进、批量推进与跨窗口推进共用同一套判断结算。
- [x] 判断只在对应 fixture 实际完成后结算；未开赛、延期或不在当前结果集中的判断不得被静默清空。
- [x] 同一场比赛只能结算一次，重复点击、快速推进和页面恢复不得重复记录。
- [x] 结算使用加时后的比赛胜负，并为点球大战和仍允许平局的赛事定义清晰语义。
- [x] 单轮推进、批量推进、推进到杯赛和推进到赛季末获得相同结果。
- [x] 加入固定种子测试，覆盖三类判断、命中、未命中、平局、加时、点球、跨窗口和批量推进。

### 2.2 推进反馈与并发保护

- [x] 保持所有推进入口拒绝并发执行，20 次快速触发仍只接受一次有效操作。
- [x] 操作触发后先渲染明确的进行中状态，再执行同步模拟计算。
- [ ] 推进失败时恢复可操作状态并显示用户可理解的错误反馈，不只写控制台。
- [x] 推进完成后优先呈现关注球队结果、观察判断结算和关键故事变化。
- [x] 保持现有 Worker 存档、页面隐藏持久化和旧写入防覆盖能力。

### 2.3 上帝之手语义

- [x] 明确“上帝之手”是可选命运实验，不是默认核心循环。
- [x] 使用前说明其影响的是球队长期基础属性，而不是单场临时状态。
- [x] 使用后在世界档案中留下“该宇宙曾被干预”的可见标记。
- [x] 保证一次性限制、赛季重置和存档恢复语义一致。
- [ ] 评估是否仅在沙盒/实验规则下默认展示；自由观察模式不得强迫玩家注意它。
- [x] 不实现分支世界线、回溯撤销或多存档树。

## 3. P0 前五分钟入口与观察焦点

### 3.1 推荐体验入口

- [x] 将首次入口区分为 `推荐体验` 与 `自选宇宙`，不移除现有自由设置能力。
- [x] 推荐体验使用经过自动审计的自然种子，而不是修改引擎或写死比赛结果。
- [x] 至少审计 20 个候选种子的前 6 个窗口，选择能稳定展示焦点比赛、爆冷可能和故事变化的推荐种子。
- [x] 推荐种子仍可公开、复制和复现，保持“同种子同宇宙”的承诺。
- [x] `自选宇宙` 继续提供玩法模式、关注球队、随机种子和球队编辑器入口。
- [x] 游戏模式与种子等高级选项默认折叠，避免首次进入时形成过多抽象选择。
- [x] 不制作新的营销落地页；第一屏直接服务于开始观察。

### 3.2 观察焦点

- [x] 在“最多三支关注球队”之上定义一个明确的主要观察球队，避免三个关注对象争夺首屏权重。
- [x] 允许不选择球队并以纯观察者身份进入。
- [x] 允许在设置中更换主要观察球队；更换不改变历史结果和已有故事记录。
- [ ] 提供 `帮我选择`，根据豪门、黑马、草根三类叙事倾向推荐球队。
- [ ] Dashboard 第一屏优先显示主要观察球队的下一场比赛、近期状态和当前故事。
- [ ] 次要关注球队保留快捷入口和摘要，但不复制主要观察模块。

### 3.3 观察主题

- [ ] 将“赛季目标”命名为轻量的“本赛季观察主题”，避免经理任务感。
- [ ] 第一版主题仅包含：豪门守成、黑马挑战、升级/保级、球员成长、纯观察。
- [ ] 根据球队级别、实力、预期排名和球员年龄自动推荐合理主题。
- [ ] 主题只负责内容排序、进度表达和赛季回顾，不提供球队能力加成。
- [ ] 主题失败不惩罚玩家，不阻止推进，也不产生资源负债。
- [ ] 主题进度从真实积分榜、赛事状态和球员数据派生，不能维护第二套数字。
- [ ] 新赛季允许继续、重选或关闭主题。

### 3.4 Dashboard 焦点模块

- [ ] 首屏只保留一个“本轮观察焦点”，统一下一场比赛、主题进度、关键球员和主要操作。
- [ ] 焦点比赛优先级依次考虑：主要观察球队、决赛/淘汰赛、争冠/保级关键战、德比、强强对话。
- [ ] 同一场比赛不得在焦点卡、强强对话通知和新闻提示中重复出现。
- [ ] 主要操作根据状态明确显示 `做出判断`、`观看比赛`、`揭晓结果` 或 `继续观察`。
- [ ] 全局数据、普通新闻和次要赛事继续存在，但不得抢占首个移动端视口。

## 5. P0 真实比赛解释与“命运偏差”

### 5.1 结构化赛前因素

- [x] 定义结构化 `MatchFactor`，至少包含来源、受益方、方向、重要度、公开标签和证据值。
- [x] 从与模拟共用的比赛模型派生因素，不在 UI 中重新估算一套强弱逻辑。
- [x] 覆盖球队基础能力、可用阵容、伤停损失、士气、疲劳、势头、主客场、教练和赛事适性。
- [x] 每场最多显示三项最有区分度的因素；接近零的因素不占位置。
- [x] 同类因素合并，避免同时出现三条都在描述阵容实力。
- [x] 因素文案使用区间和自然语言，不公开足以让玩家反推完整公式的内部小数。
- [x] 预测和正式模拟对同一 fixture、阵容与窗口使用完全一致的输入快照。

### 5.2 赛后转折点

- [ ] 从真实事件中识别红牌、伤退、领先被逆转、绝平、绝杀、加时和点球大战。
- [x] 每场最多展示两个转折点；没有明确转折时诚实显示“结果主要来自整体实力与随机波动”。
- [x] 不把比赛结束后随机生成的控球率、射门数当作赛果原因。
- [x] 赛后解释区分“赛前条件”“比赛事件”和“随机偏离”，避免混成一句万能文案。
- [x] 球员姓名、事件时间、比分变化与 MatchResult 事件完全一致。

> 当前剩余：比赛结果事件没有实时伤退类型；红牌、逆转、绝平、绝杀、加时和点球大战均已接入。后续若让伤退成为比赛内事件，必须由模拟器生成真实球员与分钟后再勾选，不从赛后伤病抽签倒推。

### 5.3 命运偏差

- [x] 定义单一、可测试的命运偏差指标，用于表达实际结果相对赛前分布的意外程度。
- [x] 指标至少区分正常结果、轻微意外、明显爆冷和重大爆冷。
- [x] 平局、加时和点球大战拥有明确且稳定的计算规则。
- [x] `ResultAnimation`、比赛详情、新闻和历史经典比赛共用同一爆冷定义。
- [x] 不将命运偏差伪装为精确因果概率；提供一句易懂说明。
- [x] 固定种子回归测试覆盖强队获胜、弱队获胜、强弱平局和淘汰赛结果。

## 6. P1 连续故事导演

### 6.1 最小故事模型

- [ ] 定义轻量 `Storyline`，只保存故事类型、参与对象、开始窗口、当前阶段、证据、最近更新和结局。
- [ ] 故事阶段限定为 `出现 / 发展 / 高潮 / 落幕`，不建设通用剧情脚本语言。
- [x] 每个世界最多同时突出一条主要观察故事和一条全局故事。
- [ ] 同一球队同类故事设置冷却和迟滞条件，避免一轮出现、下一轮消失。
- [x] 故事检测完全确定性；AI 未来最多润色文案，不负责判断故事是否发生。
- [ ] 活跃与已结束故事有明确存储上限，保持长期存档预算。

### 6.2 第一批三类故事

- [ ] 黑马崛起：结合赛季前预期、球队实力、当前排名、积分差和关键胜利。
- [ ] 豪门危机：结合声望/预期、排名偏差、连败、教练压力和关键赛事出局。
- [ ] 升班马求生：结合上赛季升级记录、保级线、阶段排名和最终结果。
- [ ] 每类故事都定义触发、升级、降温、成功结局和失败结局。
- [ ] 每类故事至少准备 6 个固定种子或构造场景测试，覆盖边界条件和赛季结束。
- [ ] 暂缓黄金一代、宿敌复仇、复杂球员传记和多球队交叉剧情。

### 6.3 故事呈现

- [x] Dashboard 只显示故事当前章节、证据和下一观察点，不展示长篇小说。
- [ ] 新闻系统在故事升级或结局时提升优先级，普通轮次不重复播报同一句话。
- [ ] 关键比赛卡显示它与当前故事的关系，例如“保级转折战”或“黑马试金石”。
- [ ] 赛季回顾总结故事起点、关键节点和结局。
- [ ] Chronicle/History 只增加紧凑故事时间线，不进行大规模职责重写。
- [x] 故事文案始终可由数据证明，避免使用“管理层震怒”等无来源事实。

- [ ] 主要观察球队发生变化后，过去赛季的关注记录保持原样。
